## 前言

按照计划，正式着手简历项目的开发。第一步就是此课程资料问答系统的原始文件准备和文件加载层的实现。

## 学习内容
### 课程资料数据设计

我选取了《编译原理》和《计算机网络》课程的相关资料，文件类型涵盖了`pdf、markdown、docx、png、jpg`，按照计划实现分阶段开发，即优先实现最基础的功能，跑通整个系统，v2阶段再增加关于图片、md内联图片、低文本pdf相关资料的加载功能，所以在数据加载之前，首先用一个脚本对所有的原始数据文件进行预处理，在一个清单`manifest.json`文件中记录一些特征，也便于实现后续功能对于文件的读取功能。比如下面几个重要的特征：
- doc_id：文档id，唯一标识符；
- course：课程名称；
- file_type：文档类型；
- priority：优先级，分为mvp、v2和skip（跳过，涉及到隐私等）；
- parse_strategy：文档加载处理策略，下面会详细介绍；

此阶段亮点总结如下：
```
实现了多课程、多格式真实数据的预处理，隔离隐私数据与公开样例，manifest驱动的数据加载策略，图片感知md，为后续多模态扩展预留空间。
```

### 数据加载

对原始的课程资料文件做完“标记”之后，便开始实现数据加载层，即**奖各种非格式化的文档转换为程序可处理的结构化数据**，这涉及到后续索引构建、检索乃至生成结果的质量。MVP阶段，本系统主要将原始文档提取为可处理的纯文本（这里是md），最终**将异构文本转换为统一的数据结构**，方便后续分块`chunk`操作。

具体到本项目，数据加载层基于前置的manifest文件，根据文件类型、解析策略等属性输出统一的`LoadedDocument`。先说明整体的文档解析策略：

| 数据类型 | 默认解析方式 | 原因 |
| --- | --- | --- |
| Markdown 课堂笔记 | 原生读取 `markdown_native` | 保留标题、列表、图片引用，最稳定 |
| 文本型 PDF 课件 | `pypdf` 按页抽取 `pdf_text_layer` | 内容覆盖完整，速度快，天然保留页码 citation |
| DOCX / 复杂文档 | Docling `docling_document` | 能输出 Markdown/JSON，结构化能力更强 |
| 低文本 PDF / 扫描件 | v2：OCR 或 Docling OCR | 当前不阻塞 MVP |
| Markdown 中引用的图片 / 单独图片 | v2：OCR 或 VLM caption | 需要结合上下文处理，不适合孤立入库 |
| 表格密集页 / 图示密集页 | v2：专项解析 | 需要质量评估后按需增强 |

选择上述文档解析策略的核心目标是（优先级从高到低）：`文本内容完整性、pdf页码可追踪、结构保留、多模态增强`。初步计划后续再增加对图片、少文本pdf页面的加载功能，首先是要跑通一个完整的RAG主链路。

对于课件pdf，其实一开始是尝试直接使用Docling，但在测试时发现，受限于我本地CPU资源有限以及内存有限（PS：该换电脑了），处理到后面的时候页面的`PDF pipeline`申请内存直接失败了，所以暂时用`pypdf`处理，从目前的效果来看，其可以按照pdf页码来提取文本，但是有一个问题：很多页面换行问题严重，甚至出现了几个字为一行的情形。为了不影响后续分块、向量化等阶段的效果，所以加了**文本归一化**，合并短行。
> 后面又发现，之前codex写的逻辑是把整个pdf给塞进Docling处理，不爆内存才怪。所以加了v2阶段新方法：用 Docling 的 page_range 参数逐页或小批量调用，生成最终的LoadedDocument。

最终，数据加载层输出一个`LoadedDocument`，统一数据结构，提供加载结果给后面的`chunk`层。
