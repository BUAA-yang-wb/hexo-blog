## 前言

开始学习Agent相关知识了。由于时间比较紧张，所以直接读一个开源项目：[Mini版Claude Code](https://github.com/Windy3f3f3f3f/claude-code-from-scratch)，其将CC的源码压缩了很多，保留了最核心的架构设计等。可能本项目难免漏掉一些知识，但足以对一个优秀的Agent有基础的认识。

## 学习内容
### 建立整体概念

Agent本质上是**模型决定下一步做什么，代码中只提供循环框架和工具**，整个Agent架构实际上就是一个`while(True)`循环：

```
while (true) {
    调用模型 → 模型返回响应
    if (响应包含工具调用) → 执行工具 → 把结果喂回模型 → 继续循环
    if (响应只是文本) → 任务完成，退出循环
}

```

其中包含几大模块：**核心Agent Loop、工具系统、上下文工程、系统提示词、权限安全**。

### Agent Loop

主体就是一个循环，直到LLM决定不适用用工具的时候就跳出循环。

先简单了解`Claude Code`是怎么处理的。它将Agent Loop拆成了两层：
- **QueryEngine：** 会话级，即某一次对话的整个生命周期，主要负责用户输入处理、预算检查、Token统计、会话恢复；
- **queryLoop：** 单轮级，管理某次输入的执行，负责消息压缩、API调用、工具执行】错误恢复；

源码中在七种条件下循环会继续，此简化代码只保留了第一种，也是最核心的：**模型决定tool_use，则继续，否则退出**。上述两层循环还有一个机制：错误扣留，即当`queryLoop`遇到错误，会先尝试执行恢复逻辑，若成功则用户感知不到，失败则暴露给`QueryEngine`。实际上，绝大多数错误都可以通过错误恢复机制解决，比如超过最大token输出、提示词过长。

关于工具执行，CC在API流式响应期间是并行执行工具的，即某个工具的JSON解析完成则单独执行。

在整个循环期间，本质上就是消息数组不断增长的过程，在用户第一次输入问题之后，会往数组中写入三条消息：
```
messages = [
    { role: "user",      content: "帮我修复 bug" }
    { role: "assistant", content: [text + tool_use(read_file)] }
    { role: "user",      content: [tool_result("文件内容...")] }
  ]
```

此后每轮对话都会往数组中增加两条：**assistant执行工具调用、user工具调用结果**，直到最后一轮模型判断不再使用工具。

### 工具系统

工具系统核心流程为：**LLM响应->生成tool_use_block->分发tool->执行tool->返回执行结果**。在工具执行的周期中，有以下几个阶段需要注意：
- 对输入进行两阶段验证：字段类型检查和业务逻辑检查，比如编辑文件tool需要检查old_string是否唯一；
- 对于工具执行结果，若超过设定的最大字符数，则将完整结果保存到磁盘中，而模型收到截断提示符通过FileReadTool主动拉取需要的结果。

执行阶段，非并发安全的工具需独占执行，而多个并发安全的工具同时执行，从而降低总执行时长。

**edit_file_tool:**
其执行前有14步验证，按照I/O成本排序，最关键的有三个：
- **读取前置检查：** 代码层面的约束，在编辑前必须先读取文件，防止模型基于过时记忆进行修改从而报错；
- **外部修改检测：** 在修改前通过mtime检测文件是否被外部修改（因为在读取和真正修改之间有空挡，这中间用户有可能会修改），mtime未变化则安全，否则重新读取。
- **配置文件保护：** 对一些项目中的配置文件，修改后会先模拟执行并进行`JSON Schema`校验，防止破坏配置格式。

**search-and-replace:**
对于修改文件内容，这里的选择是搜索然后替换，这能防止LLM出现幻觉，比如替换一个当前文件中不存在的字符串，如果实施全文件重写的话，不仅会编写错误，还会浪费大量token。
> 关于如何搜索，后面会有介绍。

**Diff输出机制：**
在搜索匹配成功后返回文件中要修改的原始字符串，并按照其风格进行修改替换，最后输出修改前后对比。

**write_file:**
系统提示词规定了优先用edit_file，只对新文件才执行write_file。

**grep_search:**
CC用的是`ripgrep`，而我们用的系统的`grep`，前者是后者的改进版，对于代码库文本匹配效果更好更快。

**结果截断：**
前面提到了若工具执行结果超过了最大字符数，给LLM返回的结果会进行截断。这儿**保留结果头尾，并告知LLM发生截断，让其决定是否需要获取完整结果内容**。

**WebFetch：**
让Agent能访问URL并获取网页信息、查文档、读API响应等。有以下设计：
- 超时：设定指定时间，超时还未获取到内容则停止，防止阻塞；
- 并行安全：此工具为只读，可并行执行；
- HTML去标签：抓取到的是HTML网页内容的话，将一些sytle、script等替换为`""`，避免浪费token；
- 上限：设定网页内容上限，防止过多占用上下文；

**ToolSearch延迟加载**
当Agent中工具较多时，给LLMapi只发送常用工具的`schema`，不常用工具只发送名称，让LLM自己判断是否需要，若需要则使用`ToolSearch`工具进行激活。
