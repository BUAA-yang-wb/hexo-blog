## 前言

今天继续学习CC中的其他模块，主要是以下几个：
- System Prompt 工程
- REPL交互式命令行实现
- 流式输出与双后端
- 权限与安全模块
- 上下文管理

## 学习内容
### System Prompt 工程

整体结构为：**有一个预设的系统提示此模板，每轮对话会将当前工作目录、git信息、claude.md信息、记忆系统、技能描述、agent描述等填进模板，并最终传给APIsystem参数**。具体分为七层递进结构，**先建立身份和约束框架，再填充具体行为指导**：
```
1. Identity   → 我是谁？interactive agent
2. System     → 运行环境的基本事实
3. Doing Tasks → 怎么写代码？（反模式接种）
4. Actions    → 哪些操作需要确认？（爆炸半径框架）
5. Using Tools → 怎么用工具？（偏好映射表）
6. Tone & Style → 输出什么格式？
7. Output Efficiency → 怎么更简洁？
```

下面介绍其中几个重要的模块：

#### 反模式接种

即明确告诉模型“不要做什么”，比描述“要做什么”有效，因为LLM会对正面指令进行“扩充”，可能会产出一些并不像要的结果，比如“让代码简洁”，LLM可能会认为“加注释会通俗易懂，从而简洁易读”。源码中有以下三个“不要”：
- 不要扩大范围：修bug时不重构周围代码；
- 不要防御性编程：不为不可能发生的场景添加`try-catch`和校验；
- 不要过早抽象：三行相似的代码比一个抽象更好；

#### 爆炸半径框架

给LLM一个**风险评估框架**，CC使用的是两方面：**可逆性+影响范围**，比如`force push`、删除云资源等操作既不可逆（即不能恢复），也影响共享环境（github仓库等），而编辑文件操作可逆且只影响本地）。此过程没有穷举一些规则，这给了LLM很大的扩展性，能快速贴合新场景。

另外，用户批准了一次操作，不代表后续类似的操作也被批准，既授权范围限制在每一次对话。

#### 工具映射表

在执行读写文件、搜索、匹配等操作时，规定LLM使用agent中的专属工具而非从LLM训练数据中学习到的各种bash命令（尽管绝大部分情况下底层功能是一样的，只不过封装为了工具而已）。

#### CLAUDE.md文件

`CLAUDE.md`是项目级指令文件，CC从五个位置加载：**全局管理策略 → 用户主目录 → 项目目录（CWD 向上遍历）→ 本地文件 → 命令行指定目录**。此外，由**近因效应**，子目录的规则覆盖了父目录的相同规则。

此文件支持`@`语法引用外部文件，并且CC支持`.claude/rules/*.md`文件**自动加载**。`@`有三种路径格式：相较于当前目录`@./relative/path`、用户目录`@~/path`、绝对路径`@/absolute/path`。

### REPL交互式命令行实现

此模块负责对用户命令行输入的命令进行解析。比如执行`/clear`等命令、解析出提示词然后调用`agent.chat`等等。支持`Ctrl+C`中断，从而制止agent走偏的行为。此外，长时间运行支持流式输出日志，让用户感知agent在干什么，并且JSON会话存储是每轮追加一行而不是整体覆盖，防止出现崩溃导致全部丢失。

### 流式输出与双后端

CC支持输出结果逐字输出，并支持Anthropic和OpenAI两套API后端。选择流式输出的原因一方面是如果回答过程稍长，用户等待的时间就会长，影响体验，另一方面输出中的方向如果错了，用户可以随时终端（节省token）。

CC还有一个优化**当LLM还在生成内容时，只要解析到了tool_use_block则立即执行对应tool，大大加快了工作效率**（注意，在执行前还要判断工具是否并行安全）。另外还有错误重试机制，规定某些错误重试（比如网络问题），而某些配置问题等则不用（重试无意义）。对于重试的时间间隔采用指数退避算法（梦回计网），防止出现大规模重试。

> 对于OpenAI来说，并不像Anthropic那样自动地并行执行tool，而是将并行安全工具归入一个批次，然后显式调用`Promise.all`来执行。

**Extended Thinking：**

额外思考模式，让LLM在一个“草稿纸上”进行推理规划，分为三种模式：adaptive（某些模型自动开启，某些需要自行决定）、enabled（需要显式开启）、disabled（某些模型不支持思考）。此过程产生的上下文不塞入上下文窗口，避免大量无效信息污染。

### 权限与安全模块

整体设计实现了**写死规则+用户定义规则的结合**，agent可以自主放行一些安全操作、自主拦截危险操作，必要时让用户确认某些不确定的命令。核心思路是：**多层检查，deny优先**。

具体的，CC实现了七层防御，严格把控安全。以下是几个重要的设计细节：
- **yolo模式：** 尽管yolo模式是“跳过安全确认”，但**并不是真的绕过一切**。在用户确认前还有多层检查机制。
- **tree-sitter:** 解析命令用的是`tree-sitter`而不是正则，因为前者能理解命令语义结构，而后者只能匹配文本，灵活性太差。当tree-sitter遇到不理解的结构时会交由用户确认；
- **用户确认：** 此处有交互对话框、Hook、ML分类器竞速决定，也就是说三者同时进行判断，只有第一个决定生效。

具体到本项目，简化了此模块，只包含**危险命令检测、权限规则系统、统一权限检查、会话级白名单**四层。

### 上下文管理

上下文管理是一个很重要的模块，直接决定了单轮对话的长久性与高质量。整体上，**CC采用五层分级压缩机制，从轻量级截断到全量摘要逐级递进**。

先看**上下文构建**。每次API调用，CC会将三类信息组成进请求：
- **系统提示词：** 由归属头（即身份说明）、工具Schema、安全规则等拼接而成。分为静态和动态半区，前者全球共享缓存，节约成本；后者为用户级别；
- **系统/用户上下文：** 注入顺序的设计合理：系统上下文置于系统提示词后面，而用户上下文置于消息数组前面，从而**确保最稳定的内容在最前面，最大化缓存命中**。
- **消息历史：** 记录对话中的一切，是后续压缩管道处理的主要部分。在发给api前会进行格式修复。

下面介绍具体的五层压缩管道，核心理念是**渐进式压缩**，成本从低到高：
- **Tool Result 预算裁剪：** 工具执行结果若超过50K字符，则持久化到磁盘，上下文中只保留紧凑引用和2KB预览。LLM可随时根据需要使用READ工具读取完整结果；
- **History Snip：** 裁剪掉历史中的冗余部分。比如同一个文件被READ多次，只保留最新的；
- **Microcompact：** 清理不再需要的旧工具结果，分为缓存冷（超过N分钟）和缓存热：前者直接修改消息内容；后者删除远程api服务端的内容，不修改本地消息，避免缓存不命中；
- **Context Collapse：** 折叠，不修改原始消息，而是创建一个折叠视图，只保存对后续完成任务最有用的状态。其启用时会抑制下面的autocompact，避免竞争；
- **Autocompact：** 最后的手段，fork子Agent调用API，将完整历史压缩为一份摘要，可以设定一个触发阈值（比如80%）。压缩提示词用**分析-摘要**两阶段，先让LLM推理，然后生成summary，最后只返回摘要。
